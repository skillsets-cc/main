# Maintainer Review Checklist

Use this checklist when reviewing skillset submissions. This ensures consistent quality and thorough validation across all submissions.

---

## Pre-Review: Automated Checks

Before starting manual review, verify CI has passed:

- [ ] ‚úÖ GitHub Actions workflow completed successfully
- [ ] ‚úÖ JSON Schema validation passed
- [ ] ‚úÖ Required files present (README.md, AUDIT_REPORT.md, content/)
- [ ] ‚úÖ Content structure valid (.claude/ or CLAUDE.md exists)
- [ ] ‚úÖ No files exceed 1MB
- [ ] ‚úÖ No secrets detected
- [ ] ‚úÖ Author verification passed

**If CI failed**: Request fixes before proceeding with manual review.

---

## Part 1: Production Proof Verification

This is the most critical part of the review. Production proof separates skillsets.cc from other registries.

### 1.1 Production URL Check

- [ ] Visit `verification.production_url` from `skillset.yaml`
- [ ] URL is accessible (not 404, not down)
- [ ] URL leads to one of:
  - [ ] A live deployed application/service
  - [ ] A public GitHub repository with recent activity (last 3 months)
  - [ ] A detailed case study or blog post
  - [ ] Documentation of production deployment

**Red Flags**:
- ‚ùå URL is unreachable or returns errors
- ‚ùå URL leads to personal homepage with no project details
- ‚ùå URL is a placeholder or "coming soon" page
- ‚ùå URL has no connection to the submitted skillset

### 1.2 Production Proof Quality

If `verification.production_proof` file (typically `PROOF.md`) is included:

- [ ] Review screenshots/videos for authenticity
  - [ ] Screenshots appear genuine (not stock images)
  - [ ] Screenshots show the skillset in use
  - [ ] Videos demonstrate actual workflow
- [ ] Check testimonials are from verifiable sources
  - [ ] Names and titles provided
  - [ ] Linked to real people (LinkedIn, GitHub)
  - [ ] Testimonials are specific, not generic
- [ ] Verify metrics are realistic and credible
  - [ ] Numbers make sense for the claimed usage
  - [ ] Metrics match repository evidence
  - [ ] Not exaggerated or fabricated

**Red Flags**:
- ‚ùå Screenshots look staged or fake
- ‚ùå Testimonials are anonymous or vague
- ‚ùå Metrics seem inflated or unrealistic
- ‚ùå No actual evidence of production use

### 1.3 Repository Evidence (if GitHub URL)

If production_url points to a GitHub repository:

- [ ] Repository is public and accessible
- [ ] Recent commits (within last 3 months for "active" status)
- [ ] Clear evidence of Claude Code usage:
  - [ ] `.claude/` directory present
  - [ ] `CLAUDE.md` or similar instructions
  - [ ] `PROCESS_DOCS/` with design or execution files
- [ ] Not a fork without original contributions
- [ ] Has meaningful activity (not just initial commit)

**Repository Quality Indicators**:
- Stars: 10+ (good), 50+ (excellent)
- Forks: 5+ (good)
- Issues/PRs: Shows active development
- Documentation: Comprehensive README
- Tests: Test suite present and passing

### 1.4 Deployed Application Evidence (if live URL)

If production_url points to a deployed application:

- [ ] Application is live and functional
- [ ] Clear connection to the submitted skillset
  - [ ] Technology stack matches skillset description
  - [ ] Features align with claimed capabilities
- [ ] Shows signs of real usage:
  - [ ] User accounts/content (if applicable)
  - [ ] Analytics/metrics visible
  - [ ] Professional presentation
- [ ] Not a demo or staging environment (unless clearly a substantial demo)

---

## Part 2: Quality Assessment

### 2.1 README.md Quality

- [ ] **Installation Instructions**
  - [ ] CLI installation command provided
  - [ ] Manual installation steps included
  - [ ] Clear and easy to follow
- [ ] **Usage Instructions**
  - [ ] How to use each skill
  - [ ] How to invoke agents
  - [ ] Concrete examples provided
- [ ] **Configuration**
  - [ ] Customization options explained
  - [ ] Configuration examples given
- [ ] **Documentation Clarity**
  - [ ] Well-organized sections
  - [ ] Proper grammar and spelling
  - [ ] Code blocks formatted correctly
  - [ ] No broken links

**Quality Levels**:
- **Excellent**: Comprehensive, clear, with examples
- **Good**: Complete but could use more detail
- **Needs Improvement**: Missing sections or unclear

### 2.2 AUDIT_REPORT.md Review

- [ ] Audit report exists and is complete
- [ ] Shows "PASSED" status
- [ ] All required checks are marked as passed
- [ ] File inventory is accurate
- [ ] No critical issues reported
- [ ] Generated by `/audit_skillset` skill (or equivalent validation)

**If audit shows warnings**:
- [ ] Warnings are acknowledged in PR description
- [ ] Warnings are justified and acceptable

### 2.3 skillset.yaml Validation

Already validated by CI, but double-check:

- [ ] Description accurately reflects the skillset
- [ ] Tags are relevant and appropriate
  - [ ] No spam tags
  - [ ] Not too many tags (max 10)
  - [ ] Tags describe the skillset well
- [ ] Compatibility claims are realistic
- [ ] Version starts at 1.0.0 for new submissions
- [ ] Status is "active" for new submissions

### 2.4 Content Quality

Navigate to `content/` directory:

- [ ] **Skills Documentation**
  - [ ] Each skill has a `SKILL.md` or documentation file
  - [ ] Skill purpose is clear
  - [ ] Usage examples provided
  - [ ] Expected inputs/outputs defined
- [ ] **Agents Documentation**
  - [ ] Each agent has documentation
  - [ ] Agent responsibilities clear
  - [ ] When to use each agent explained
- [ ] **CLAUDE.md or .claude/**
  - [ ] Project-level instructions present
  - [ ] Clear workflow explained
  - [ ] Integration instructions provided

**Code Quality** (if applicable):
- [ ] No obvious code smells
- [ ] Follows common conventions
- [ ] No hardcoded secrets or credentials
- [ ] Error handling present

---

## Part 3: Originality and Value

### 3.1 Uniqueness Assessment

- [ ] Check if similar skillsets already exist
- [ ] If similar, does this offer something new?
  - [ ] Different approach or methodology
  - [ ] Better documentation
  - [ ] More comprehensive
  - [ ] Different tech stack or use case

**Note**: Some overlap is okay if the skillset adds value.

### 3.2 Value Proposition

- [ ] Clear problem statement
- [ ] Solves a real need
- [ ] Would be useful to other developers
- [ ] Not overly specific to one project (unless exemplary)
- [ ] Reasonable scope (not too narrow, not too broad)

### 3.3 Completeness

- [ ] Skillset feels "complete"
  - [ ] Not missing obvious components
  - [ ] Workflow makes sense end-to-end
  - [ ] No "TODO" or "coming soon" placeholders
- [ ] Ready for others to use
  - [ ] Installation would succeed
  - [ ] Documentation is sufficient
  - [ ] Dependencies are clear

## Part 4: Production Proof Quality Standards

Categorize the submission based on production proof quality:

### High Quality (Fast-Track Approval) ‚≠ê‚≠ê‚≠ê

Characteristics:
- Public deployed service with verifiable usage
- Open-source repository with 10+ stars and regular activity
- Detailed case study with metrics and outcomes
- Professional presentation
- Strong evidence of production use

**Examples**:
- Live SaaS product built with the skillset
- Popular open-source project using the workflow
- Comprehensive blog post with before/after metrics

**Action**: Approve immediately if all other checks pass.

### Acceptable (Standard Review) ‚≠ê‚≠ê

Characteristics:
- GitHub repository with README showing real usage
- Deployed application with evidence of functionality
- Blog post or documentation of production deployment
- Screenshots/videos of working system
- Moderate evidence of production use

**Examples**:
- Personal project deployed to production
- Internal tool with screenshots
- Freelance project with client permission

**Action**: Approve if quality checks pass.

### Needs Improvement (Request More Proof) ‚≠ê

Characteristics:
- Vague or generic descriptions
- Minimal evidence of actual production use
- Dead links or inaccessible proof
- Claims not well-supported by evidence
- Unclear connection between proof and skillset

**Examples**:
- "Trust me, it's used in production"
- GitHub repo with only initial commit
- Unrelated production URL

**Action**: Request additional proof before approval.

### Insufficient (Reject or Request Major Changes) ‚ùå

Characteristics:
- No working production_url
- Fabricated or misleading proof
- Skillset has not been used in production
- Security concerns in code
- Violates community guidelines

**Examples**:
- Completely made up metrics
- Fake screenshots
- Never actually used in production
- Malicious code

**Action**: Reject with explanation.

## Part 5: Red Flags (Automatic Rejection)

If any of these are present, reject the submission:

- [ ] ‚ùå `production_url` is unreachable or fake
- [ ] ‚ùå Proof appears fabricated or misleading
- [ ] ‚ùå Skillset has clearly not been used in production
- [ ] ‚ùå Security vulnerabilities in code
- [ ] ‚ùå Plagiarism or copyright violation
- [ ] ‚ùå Spam or low-effort submission
- [ ] ‚ùå Violates Code of Conduct
- [ ] ‚ùå Contains malware or malicious code
- [ ] ‚ùå Author attempted to bypass validation
- [ ] ‚ùå Secrets or credentials exposed

- [ ] Verify site rebuild triggered

---

## Review Time Expectations

- **Simple Submissions**: 15-30 minutes
- **Complex Submissions**: 45-60 minutes
- **Submissions Needing Clarification**: Multiple rounds

**Goal**: Complete first review within 3-5 business days of submission.

---

## Tips for Efficient Reviews

1. **Start with production proof**: If it fails, save time and reject/request early
2. **Use browser dev tools**: Check for fake screenshots (reverse image search)
3. **Clone and test**: For complex skillsets, actually try installing
4. **Document patterns**: Keep notes on common issues
5. **Be consistent**: Use this checklist every time
6. **Be encouraging**: Contributors put in real work

---

## Questions or Concerns?

If uncertain about a submission:
- Ask in maintainer chat/discussion
- Request input from other maintainers
- When in doubt, request more information from contributor

---

## Maintaining Quality Standards

The registry's value comes from its curation. It's okay to:
- ‚úÖ Reject submissions that don't meet standards
- ‚úÖ Request significant improvements
- ‚úÖ Set a high bar for production proof
- ‚úÖ Take time to review thoroughly

The goal is a registry of truly production-verified, high-quality skillsets‚Äînot maximum quantity.
